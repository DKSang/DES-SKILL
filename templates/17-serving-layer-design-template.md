# Serving Layer Design

This template is used during Phase 17 (Serving Layer Design) to design the interface between data assets and end consumers — covering delivery pattern, performance targets, access control, and caching strategies.

---

## 1. Serving Pattern Selection

Choose and justify the delivery pattern for each consumer group:

| Consumer | Dataset / Model | Serving Pattern | Justification |
| :--- | :--- | :--- | :--- |
| BI Analysts | `fact_orders`, `dim_*` | **Pull** — Direct Lake / Import via Power BI | Analysts use self-serve; low latency not required |
| Operations Team | Real-time KPIs | **Pull** — DirectQuery via Power BI | Requires up-to-the-minute data; accepts slower query time |
| Mobile App | Customer account summary | **Push** — REST API (FastAPI + Redis cache) | Low-latency API response < 200ms required |
| CRM System | Updated customer segment | **Reverse ETL** — Census / Hightouch | Operational system needs data pushed back proactively |
| ML Feature Store | Training features | **Pull** — JDBC batch read | Daily batch sync; no real-time requirement |
| External Partners | Order summary reports | **Pull** — Scheduled SFTP file export | Partners have no direct data platform access |

## 2. Performance & SLA Requirements

| Consumer / Channel | Latency SLA | Throughput | Data Freshness SLA | Uptime SLA |
| :--- | :--- | :--- | :--- | :--- |
| Power BI Import | < 30s dashboard load | 200 concurrent users | Refreshed by 7AM daily | 99.5% |
| REST API | < 200ms P99 response | 500 req/sec peak | Near-real-time (< 5 min lag) | 99.9% |
| Reverse ETL (CRM) | < 1 hour end-to-end | 1M records/sync | Updated by 8AM daily | 99.5% |
| SFTP Export | < 30 min to generate | Weekly file < 500MB | Generated by 6AM Monday | 99% |

---

## 3. Caching & Materialization Strategy

*   **BI Import Cache**: (e.g., Power BI dataset imports full Gold layer at 7AM. Cache valid until next scheduled refresh.)
*   **API Response Cache**: (e.g., Redis cache TTL = 5 minutes for customer summary endpoint. Invalidated on CRM update events.)
*   **Pre-computed Aggregate Tables**: (e.g., `gold.daily_revenue_by_region` pre-aggregated at midnight to avoid full-scan queries at 9AM peak)
*   **DirectQuery Optimization**: (e.g., Clustered by `order_date + region` to enable partition pruning in DirectQuery mode)

---

## 4. Access Control & Authentication

| Consumer | Auth Method | Permission Level | Row-Level Filter |
| :--- | :--- | :--- | :--- |
| Power BI workspace | AAD OAuth / Service Principal | Read — Gold layer only | Regional RLS applied per user |
| REST API | API Key + JWT Bearer token | Read — specific endpoints only | Customer can only access own records |
| ML Feature Store | Service Account + managed identity | Read — Silver + Gold | No row filter |
| SFTP Export | SSH key + IP whitelist | Read — export bucket only | Pre-filtered by partner_id |

---

## 5. Delivery Format & Schema

*   **BI**: Star schema (Fact + Dimensions) published via semantic model. No raw Bronze/Silver tables exposed.
*   **API**: JSON response. Schema versioned (`/v1/`, `/v2/`). OpenAPI spec published to API gateway.
*   **File Export**: Parquet or CSV with explicit schema header. Encrypted at rest (AES-256). Signed with SHA-256 checksum file.
*   **Reverse ETL**: Field mapping defined in Census/Hightouch config. Only non-PII fields synced to CRM unless explicit consent is granted.

---

## 6. Observability & SLA Monitoring

*   **API Health Check**: (e.g., Synthetic probes every 60 seconds from monitoring system. Alert if P99 > 500ms.)
*   **BI Refresh Monitor**: (e.g., Airflow sensor confirms Power BI dataset refresh completes before 8AM.)
*   **Reverse ETL Sync Monitor**: (e.g., Census sync logs checked after each run. Alert if > 1% of records fail to sync.)
*   **Escalation Path**: (e.g., Consumer-facing SLA breach → immediate Slack alert to `#data-serving-ops` → PagerDuty page if unresolved in 15 min)
