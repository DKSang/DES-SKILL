# Fundamentals of Data Engineering -> DES-SKILL Chapter Map

This document establishes the official mapping between the chapters of *Fundamentals of Data Engineering* (Reis & Housley) and the corresponding phases, skills, and templates of the `DES-SKILL` framework.

---

| Book Topic / Chapter | DES-SKILL Area / Phase | Related Skill(s) | Target Improvement Idea |
| :--- | :--- | :--- | :--- |
| **Ch 1: Data Engineering Described**<br>- Definition of Data Engineering<br>- Data Maturity Levels (1, 2, 3)<br>- Business/Technical Roles | **01: Business Discovery**<br>**22: Project Evaluation** | `de-business-discovery`<br>`de-project-evaluation` | - Add **Data Maturity Assessment** checklist to business discovery.<br>- Enforce exit gates based on business value metrics rather than tool choices. |
| **Ch 2: The Data Engineering Lifecycle**<br>- Source Systems to Serving<br>- Major Undercurrents (Security, DataOps, FinOps, Orchestration) | **End-to-End Coordination** | `using-des-skill`<br>`DES-WORKFLOW.md` | - Define strict **Phase Gates** (entry/exit criteria) for the workflow.<br>- Integrate the **Six Undercurrents** as mandatory verification checks in status files. |
| **Ch 3: Designing Good Data Architecture**<br>- Architectural Principles (coupling, reversibility, failover)<br>- Warehouses, Lakes, Mesh | **07: Architecture Design** | `de-architecture-design` | - Introduce a **Storage & Compute Tradeoff Matrix**.<br>- Require a **Decision Record Template** that documents "Reversible vs. Irreversible" architectural decisions. |
| **Ch 4: Choosing Technologies**<br>- FinOps / TCO / Opportunity Cost<br>- Build vs. Buy, Modular vs. Monolith | **07: Architecture Design** | `de-architecture-design` | - Add **FinOps & Total Cost of Ownership (TCO) calculator guidelines**.<br>- Add modularity vs. distributed monolith decision guides. |
| **Ch 5: Data Generation in Source Systems**<br>- OLTP databases, APIs, Logs, Messages<br>- Types of Time (Event vs. System vs. Ingestion) | **05: Data Source Assessment** | `de-data-source-assessment` | - Add a **Source System Behavior Profile Template**.<br>- Define checklists for upstream source team communication, SLAs, and change-management agreements. |
| **Ch 6: Storage**<br>- Disk, SSD, Objects, Files, Streaming<br>- Indexes, Partitioning, Clustering | **06: Domain Modeling**<br>**09: Bronze Layer Design** | `de-domain-modeling`<br>`de-bronze-layer-design` | - Design **Partitioning & Clustering Rules** (based on query access patterns).<br>- Define standard raw retention and serialization formats (Parquet vs. Iceberg). |
| **Ch 7: Ingestion**<br>- Batch vs. Streaming vs. CDC<br>- Latency, Watermarking, DLQs | **08: Ingestion Design** | `de-ingestion-design` | - Create an **Ingestion Pattern Decision Guide** (Batch, Streaming, CDC, File Landing).<br>- Mandate explicit **Failure Mode, Retry, and Dead-Letter Queue (DLQ)** plans in templates. |
| **Ch 8: Queries, Modeling, & Transformation**<br>- Modeling (Kimball, Inmon, One Big Table)<br>- Batch/Stream Transformations | **10: Silver Layer Design**<br>**11: Gold Layer Design**<br>**13: Transformation Design** | `de-silver-layer-design`<br>`de-gold-layer-design`<br>`de-transformation-design` | - Add **Standardization Guidelines** (handling nulls, deduping, timestamp normalization).<br>- Add Kimball dimensional modeling checklists for analytical Gold layers. |
| **Ch 9: Serving Data**<br>- Analytics, ML, Reverse ETL<br>- Semantic Layers, Query Federation | **16: Semantic Model Design**<br>**17: Serving Layer Design** | `de-semantic-model-design`<br>`de-serving-layer-design` | - Establish **Reverse ETL / Push patterns** for operational systems.<br>- Define trust metrics, access latency SLAs, and Metrics Layer specifications. |
| **Undercurrents: Security & Governance**<br>- Access Control, PII masking, Auditing<br>- Metadata Catalogs, Lineage | **14: Data Quality**<br>**18: Lineage & Metadata**<br>**19: Governance & Security** | `de-data-quality`<br>`de-lineage-and-metadata`<br>`de-governance-and-security` | - Expand **PII Transformation Guidelines** (hashing, encryption, redaction).<br>- Define column-level lineage templates and data quality anomaly detection checks. |
| **Undercurrents: Orchestration & Observability**<br>- DAGs, Scheduling, Retries<br>- Monitoring, Logging, Alerting | **15: Orchestration & Obs.** | `de-orchestration-and-observability` | - Define monitoring thresholds and alerting escalation rules.<br>- Include retry policies with backoff and dependency management design guidelines. |
